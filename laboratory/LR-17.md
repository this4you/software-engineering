# Лабораторна робота: CI/CD та автоматизація в GitHub Actions для React + Vite 🚀

## Тема
CI/CD та автоматизація процесів розробки за допомогою **GitHub Actions**.

## Мета роботи
Навчитися:
- створювати workflow у **GitHub Actions**;
- автоматизувати перевірку якості коду;
- налаштовувати базовий **CI/CD pipeline** для frontend-проєкту;
- працювати з релізами, тегами, оновленням документації та деплоєм;
- інтегрувати зовнішній API у процес автоматизації.

---

## Вхідні дані
Викладач надає студенту [готовий репозиторій](https://github.com/this4you/ci-cd-lr-project) із базовим **React-проєктом, створеним за допомогою Vite**.

Завдання студента — скопіювати репозиторій у власний GitHub-акаунт та реалізувати в ньому автоматизацію через **GitHub Actions**.

---

## Опис завдання
Необхідно побудувати **CI/CD pipeline** для React + Vite застосунку.

Студент має створити один або декілька workflow у папці:

```text
.github/workflows/
```

Pipeline повинен автоматизувати повний цикл перевірки, релізу, оновлення документації та деплою.

---

## Функціональні вимоги

### 1. Перевірка коду (CI)
Workflow повинен запускатися при:
- `push`
- `pull_request`

У межах цього workflow потрібно реалізувати:
- встановлення залежностей;
- запуск **lint**;
- запуск **тестів**;
- запуск **build**.

### Умова
Якщо **lint**, **tests** або **build** завершуються з помилкою, workflow має завершуватися зі статусом **failed**.

Іншими словами:
- неякісний код не повинен проходити далі;
- pipeline має “завалюватися”, якщо проєкт не відповідає вимогам якості.

---

### 2. Автоматичне створення релізу
Після успішного злиття Pull Request у гілку `main` потрібно автоматично:
- визначити нову версію релізу;
- створити **git tag**;
- створити **GitHub Release**.

### Примітка
Спосіб формування версії студент обирає самостійно.

Допускаються, наприклад, такі підходи:
- `v1.0.<номер запуску workflow>`
- `v<github.run_number>`
- версія з `package.json`
- будь-який інший логічний варіант

---

### 3. Автоматичне оновлення README.md
Після успішного merge Pull Request у `main` workflow повинен автоматично оновити файл `README.md`.

У README потрібно додати або оновити блок з інформацією про останній реліз:

- номер останньої версії;
- назву Pull Request;
- текст жарту, отриманий із API: `https://api.chucknorris.io/`

### Обов’язкова умова
Жарт має бути отриманий автоматично під час виконання workflow через зовнішній API.

### Приклад блоку в README
```md
## Latest Release Info
- Version: v1.0.7
- PR Title: Add responsive navbar
- Chuck Norris Joke: Chuck Norris can deploy to production on Friday and nothing breaks.
```

---

### 4. Додавання жарту в коментар
Під час виконання automation потрібно додати **коментар** з випадковим жартом.

Коментар може бути:
- у Pull Request;
- у коміті;
- у Release notes;
- або в іншому логічному місці GitHub-автоматизації.

### Джерело жарту
Потрібно використати API:

```text
https://api.chucknorris.io/
```

### Умова
Жарт має підтягуватись динамічно, а не бути записаним вручну.

---

### 5. Deploy на GitHub Pages
Після успішного проходження всіх перевірок застосунок має бути автоматично задеплоєний на **GitHub Pages**.

### Очікуваний результат
Після merge у `main` користувач повинен мати доступ до актуальної версії застосунку за публічним URL GitHub Pages.

---

## Рекомендована структура workflow
Студент може реалізувати рішення:
- або в **одному великому workflow**;
- або розділити логіку на **кілька workflow**.

### Рекомендований варіант
1. **ci.yml** — перевірка lint/test/build
2. **release.yml** — створення тегу, релізу, оновлення README, отримання жарту
3. **deploy.yml** — деплой на GitHub Pages

---

## Технічні вимоги
У роботі бажано використати:
- `actions/checkout`
- `actions/setup-node`
- кешування залежностей
- `GITHUB_TOKEN`
- умовний запуск job або step
- змінні середовища
- роботу з event payload GitHub Actions
- bash/script-команди для оновлення README або створення тегу

---

## Вимоги до реалізації
У репозиторії студента повинні бути:

1. Робочі workflow-файли у `.github/workflows/`
2. Оновлений `README.md`
3. Створений хоча б один реліз
4. Створений хоча б один тег
5. Налаштований GitHub Pages deploy
6. Тести та lint, які реально запускаються

---

## Що потрібно продемонструвати
Під час захисту студент має пояснити:

- які workflow були створені;
- на які події GitHub вони реагують;
- що виконує кожен workflow;
- як формується номер версії;
- як оновлюється README;
- як отримується жарт із зовнішнього API;
- як відбувається деплой на GitHub Pages;
- що станеться, якщо lint або tests не пройдуть.

---

## Що потрібно здати
Студент повинен надати:

- посилання на GitHub-репозиторій;
- посилання на GitHub Pages;
- посилання на створений Release;
- короткий опис реалізації в `README.md`.

---

## Критерії оцінювання

### 1. Базовий CI workflow — 1 балів
- workflow запускається на `push` і `pull_request`;
- коректно виконує встановлення залежностей;
- запускає lint і tests.

### 2. Коректне падіння pipeline при помилках — 1 балів
- при помилках у lint або tests workflow завершується зі статусом failed.

### 3. Автоматичне створення tag і GitHub Release — 1 балів
- після merge у `main` створюється новий тег;
- створюється реліз у GitHub.

### 4. Автоматичне оновлення README.md — 1 балів
- у README додається актуальна інформація про реліз;
- виводиться назва PR;
- додається жарт.

### 5. Інтеграція з Chuck Norris API — 1 балів
- жарт отримується автоматично;
- використовується зовнішній API.

### 6. Deploy на GitHub Pages — 1 балів
- застосунок автоматично публікується;
- посилання на Pages працює.

**Усього: 6 балів**

## Підказки
Під час виконання лабораторної варто звернути увагу на такі моменти:

- GitHub Release і git tag — це не зовсім одне й те саме;
- автоматичне оновлення `README.md` може випадково спричинити повторний запуск workflow;
- для деплою на GitHub Pages потрібно правильно налаштувати репозиторій;
- якщо проєкт використовує Vite, треба врахувати build output і базовий шлях застосунку;
- зчитування назви Pull Request залежить від типу події, на якій запускається workflow.

## Мінімальний очікуваний результат

- є хоча б один працездатний workflow для CI;
- є автоматичне створення релізу;
- README оновлюється автоматично;
- жарт із API використовується у workflow;
- застосунок опублікований на GitHub Pages.
