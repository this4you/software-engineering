# Лабораторна робота
# **Проєктування програмних систем: сучасні архітектурні підходи**

## Мета роботи
Навчитися **проєктувати програмну систему** на основі сучасних архітектурних підходів та реалізовувати окремий **use case** із правильним розподілом відповідальностей між компонентами системи.

У межах лабораторної роботи студент повинен не просто реалізувати функціональність, а показати вміння:
- виділяти **бізнес-логіку**;
- відокремлювати її від технічних деталей;
- будувати систему з урахуванням **чітких меж відповідальності**;
- застосовувати принципи сучасних архітектур, зокрема:
    - **Clean Architecture**
    - **Hexagonal Architecture**
    - **Onion Architecture**
    - **Ports and Adapters**

---

## Загальна постановка завдання
Кожен студент отримує **індивідуальний варіант use case**.  
Потрібно спроєктувати та реалізувати **мініпроєкт**, у якому цей use case буде реалізований із використанням принципів сучасних архітектурних підходів.

Реалізація може бути виконана **будь-якою мовою програмування** та з використанням будь-яких технологій.

Основна вимога: у проєкті повинно бути видно, що студент **усвідомлено проєктував структуру системи**, а не просто написав набір класів або функцій.

---

## Основні вимоги до роботи

Студент повинен:

### 1. Реалізувати свій use case
Для кожного варіанту необхідно реалізувати основний сценарій роботи системи.

### 2. Продемонструвати архітектурний підхід
У проєкті має бути використано один із сучасних архітектурних підходів (або комбінація підходів):
- Clean Architecture
- Hexagonal Architecture
- Onion Architecture
- Ports and Adapters

Допускається комбінування ідей, якщо це логічно обґрунтовано.

### 3. Виділити бізнес-логіку
Бізнес-логіка не повинна бути змішана з:
- кодом UI;
- кодом роботи з базою даних;
- фреймворк-залежним кодом;
- технічними деталями інтеграції.

### 4. Продемонструвати розподіл відповідальностей
У коді має бути видно окремі ролі компонентів, наприклад:
- use case / application service;
- domain model / entities;
- ports / interfaces;
- adapters / infrastructure;
- controllers / handlers / API layer.

### 5. Підготувати README
У репозиторії обов’язково має бути файл **README.md**, у якому студент повинен описати:
- який use case реалізовано;
- яку архітектуру обрано;
- чому було обрано саме її;
- як у проєкті розділено відповідальності;
- де знаходиться бізнес-логіка;
- які інтерфейси / порти / адаптери використано;
- як запустити проєкт.

---

## Що саме потрібно здати
Студент повинен здати:

- **посилання на Git-репозиторій**;
- у репозиторії мають бути:
    - файли коду;
    - файл `README.md`.

---

## Мінімальна структура проєкту
Структура може відрізнятися залежно від мови та технології, але в проєкті повинні бути присутні логічно виділені частини:

- **шар / модуль бізнес-логіки**
- **шар / модуль сценарію використання**
- **інтерфейси або порти**
- **адаптери / реалізації взаємодії із зовнішнім світом**

---

## Обов’язкові елементи реалізації
У роботі має бути:

- реалізація **основного use case**;
- хоча б **одна перевірка бізнес-правила**;
- хоча б **одна взаємодія через інтерфейс / порт / абстракцію**;
- хоча б **одна технічна реалізація адаптера**  
  (наприклад: робота з пам’яттю, файлом, mock storage, базою даних, API, console input/output, web handler тощо).

---

## Що НЕ є достатнім для виконання
Робота не вважається повноцінно виконаною, якщо:
- проєкт складається лише з controller + service + repository без пояснення архітектурної логіки;
- вся логіка зосереджена в одному файлі або одному класі;
- не видно меж між бізнес-логікою та інфраструктурою;
- відсутній README;
- README не пояснює архітектурні рішення;
- реалізовано лише CRUD без бізнес-правил і без сценарію використання.

---

## Рекомендації
Студентам рекомендується:
- почати з опису сценарію use case;
- виділити основні сутності;
- визначити бізнес-правила;
- подумати, які частини системи є внутрішніми, а які зовнішніми;
- вирішити, через які інтерфейси або порти бізнес-логіка буде працювати із зовнішніми ресурсами;
- лише після цього переходити до написання коду.

---

# Вимоги до README.md

У файлі `README.md` потрібно обов’язково описати:

## 1. Назву роботи
Наприклад:  
**“Library Book Reservation — Hexagonal Architecture”**

## 2. Опис use case
Коротко пояснити, що саме робить система.

## 3. Обрану архітектуру
Написати, який підхід використано:
- Clean Architecture
- Hexagonal Architecture
- Onion Architecture
- Ports and Adapters

## 4. Обґрунтування вибору
Пояснити, чому для цього завдання була обрана саме така архітектура.

## 5. Структуру проєкту
Описати основні пакети / модулі / директорії.

## 6. Розподіл відповідальностей
Пояснити:
- де domain logic;
- де use case;
- де adapters;
- де ports / interfaces;
- де технічна реалізація.

## 7. Інструкцію запуску
Описати, як запустити проєкт.

---

# Критерії оцінювання

Оцінюється не лише працездатність, а й **якість архітектурного мислення**.

### Під час перевірки звертається увага на:
- коректність реалізації use case;
- логічність структури проєкту;
- наявність чітких меж між шарами / модулями;
- правильне відокремлення бізнес-логіки від технічних деталей;
- доречність вибраної архітектури;
- якість README;
- зрозумілість коду та назви компонентів;
- наявність пояснення архітектурних рішень.

# Коротка версія формулювання

**Ваше завдання:** реалізувати один індивідуальний use case, використовуючи принципи сучасних архітектурних підходів: Clean Architecture, Hexagonal Architecture, Onion Architecture або Ports and Adapters.  
У репозиторії мають бути код проєкту та `README.md`, у якому пояснено, яку архітектуру ви використали, як поділено відповідальності та де розміщена бізнес-логіка.
